## 一、核心发现：MarkItDown

**GitHub**: [microsoft/markitdown](https://github.com/microsoft/markitdown)
**Stars**: 123K+
**许可证**: MIT
**Python**: 3.10+

微软 2024 年发布的文件转 Markdown 工具，是目前最成熟的开源方案。

### 支持的格式

| 格式 | 转换器 | 依赖 |
|------|--------|------|
| **DOCX** | `_docx_converter.py` | mammoth (~1.11), lxml |
| **PDF** | `_pdf_converter.py` | pdfminer.six, pdfplumber |
| **PPTX** | `_pptx_converter.py` | python-pptx |
| **XLSX** | `_xlsx_converter.py` | pandas, openpyxl |
| **XLS** | `_xlsx_converter.py` | pandas, xlrd |
| **HTML** | `_html_converter.py` | markdownify, beautifulsoup4 |
| **EPUB** | `_epub_converter.py` | 内建 |
| **CSV** | `_csv_converter.py` | pandas |
| **纯文本** | `_plain_text_converter.py` | charset-normalizer |
| **图片** | `_image_converter.py` | 内建 (EXIF) |
| **音频** | `_audio_converter.py` | pydub, SpeechRecognition |
| **YouTube** | `_youtube_converter.py` | youtube-transcript-api |
| **Wikipedia** | `_wikipedia_converter.py` | 内建 |
| **ZIP** | `_zip_converter.py` | 内建 |

### 架构亮点

1. **插件系统**：通过 `ConverterPlugin` 协议扩展，支持第三方注册新转换器
2. **MCP Server**：内置 `markitdown-mcp` 包，可作为 MCP 工具调用
3. **OCR 插件**：`markitdown-ocr` 独立包，支持 DOCX/PDF/PPTX/XLSX 的 OCR 提取
4. **Magika 文件检测**：用 Google Magika 做 MIME 类型检测，比扩展名更可靠
5. **CLI + API 双接口**：`markitdown file.pdf` 或 `MarkItDown().convert("file.pdf")`

### 与 Doc-Knowledge 的关系

MarkItDown 解决的是 A→B 转换问题，但不提供：
- ❌ 语义去重
- ❌ 价值评分
- ❌ 自动分类和标签
- ❌ 批量流水线
- ❌ 导出到 Obsidian/MemoMind

**结论：MarkItDown 是 Doc-Knowledge 转换层的理想底座，差异化能力自建。**

## 二、各格式独立方案对比

如果不用 MarkItDown 全家桶，各格式的最佳独立方案：

### DOCX → Markdown

| 方案 | 类型 | 说明 | 推荐度 |
|------|------|------|--------|
| **mammoth** | 库 | DOCX→HTML/Markdown，语义转换（不是逐字提取） | ⭐⭐⭐⭐⭐ |
| python-docx | 库 | 读取 DOCX 结构，需自行转换逻辑 | ⭐⭐⭐ |
| pandoc | CLI | 万能转换器，依赖 Haskell | ⭐⭐⭐⭐ |

**推荐：mammoth**
- MarkItDown 也用它作为 DOCX 转换器
- 语义转换优于逐字提取：`<strong>` → `**bold**`，表格自动处理
- 支持自定义样式映射
- `pip install mammoth`，无重型依赖

### PDF → Markdown

| 方案 | 类型 | 说明 | 推荐度 |
|------|------|------|--------|
| **pdfplumber** | 库 | 基于 pdfminer.six，表格提取强 | ⭐⭐⭐⭐⭐ |
| pdfminer.six | 库 | PDF 文本提取基础库 | ⭐⭐⭐⭐ |
| PyMuPDF (fitz) | 库 | 速度快，API 友好 | ⭐⭐⭐⭐ |
| Marker | AI | 深度学习 PDF→MD，支持表格/公式 | ⭐⭐⭐（重型） |

**推荐：pdfplumber（表格场景）+ pdfminer.six（纯文本场景）**
- MarkItDown 两者都用了
- 纯文本 PDF 用 pdfminer.six（更轻量）
- 含表格/布局复杂的用 pdfplumber

### PPTX → Markdown

| 方案 | 类型 | 说明 | 推荐度 |
|------|------|------|--------|
| **python-pptx** | 库 | 读取 PPTX 结构，需自行转换 | ⭐⭐⭐⭐ |
| pandoc | CLI | 支持 PPTX，但效果一般 | ⭐⭐⭐ |

**推荐：python-pptx**
- MarkItDown 也用这个
- 遍历 slide → shape → text_frame，按层级输出 Markdown
- 图片单独提取

### XLSX → Markdown

| 方案 | 类型 | 说明 | 推荐度 |
|------|------|------|--------|
| **pandas + openpyxl** | 库 | DataFrame.to_markdown() | ⭐⭐⭐⭐⭐ |
| openpyxl | 库 | 直接读取，需自行格式化 | ⭐⭐⭐ |

**推荐：pandas + openpyxl**
- MarkItDown 也这么用
- `pd.read_excel()` → `df.to_markdown()` 一行搞定
- 多 sheet 自动分节

## 三、推荐技术方案

### 方案 A：封装 MarkItDown（推荐）

```python
# 转换层直接调用 MarkItDown
from markitdown import MarkItDown

md = MarkItDown()
result = md.convert("document.pdf")
markdown_text = result.text_content
```

**优势**：
- 零维护转换逻辑，微软社区持续更新
- 14+ 格式开箱即用
- 插件系统支持扩展
- MIT 许可，商业友好

**Doc-Knowledge 自研部分**：
```
MarkItDown (转换) → 去重器 → 评分器 → 标签器 → 导出器
      ↓
   差异化能力（自研）
```

**依赖变化**：
```toml
# 原方案
dependencies = ["PyMuPDF", "python-docx", "python-pptx", "openpyxl"]

# 新方案（大幅精简）
dependencies = ["markitdown>=0.1.0"]

# OCR 可选
[project.optional-dependencies]
ocr = ["markitdown-ocr"]
```

### 方案 B：自选最佳组件（备选）

如果不想依赖 markitdown 全家桶，各选最优：

```toml
dependencies = [
    "mammoth~=1.11.0",      # DOCX
    "pdfplumber>=0.11.9",   # PDF（含表格）
    "python-pptx",           # PPTX
    "pandas",                # XLSX
    "openpyxl",              # XLSX
]
```

**优势**：更精细控制每个格式的转换行为
**劣势**：维护成本高，需要自己写 4 个转换器的完整逻辑

## 四、决策

| 维度 | 方案 A（MarkItDown） | 方案 B（自选） |
|------|---------------------|---------------|
| 开发量 | 小（封装即可） | 大（4 个转换器全写） |
| 维护成本 | 低（社区维护） | 高（自己维护） |
| 格式覆盖 | 14+ 格式 | 仅 4 种 |
| 自定义能力 | 中等（插件扩展） | 高（完全控制） |
| 成熟度 | 123K stars | 各自成熟 |

**推荐：方案 A（封装 MarkItDown）**

理由：
1. Doc-Knowledge 的核心价值在**知识提取**（去重/评分/标签），不在转换
2. MarkItDown 转换质量已经很好，没必要重复实现
3. 通过插件机制可以扩展自定义行为（如反向链接注入）
4. 保留方案 B 作为 fallback（当需要特殊定制时）

## 五、去重/评分方案

### 语义去重

| 方案 | 说明 | 推荐度 |
|------|------|--------|
| **TF-IDF + 余弦相似度** | 自建，零外部依赖 | ⭐⭐⭐⭐ |
| sentence-transformers | 深度学习嵌入，准确但重 | ⭐⭐⭐ |
| SimHash | 快速近似去重 | ⭐⭐⭐⭐ |

**推荐：TF-IDF + 余弦相似度（Phase 1）+ SimHash（Phase 2 优化性能）**

### 价值评分

| 因子 | 权重 | 说明 |
|------|------|------|
| 内容长度 | 20% | 太短（<100字）价值低 |
| 结构完整性 | 25% | 有标题/段落/列表 > 纯文本 |
| 新鲜度 | 15% | 修改时间越新分越高 |
| 关键词密度 | 20% | 专业术语密度 |
| 独特性 | 20% | 与其他文档的差异化程度 |

## 六、导出方案

### 导出到 Obsidian

直接写入 Markdown 文件 + YAML frontmatter：
```yaml
---
source: "C:/docs/原文件.docx"
created: 2026-05-17
tags: [report, technical]
score: 75
---
```

### 导出到 MemoMind

调用 MemoMind API（已有 MCP Server）：
```python
# 通过 markitdown-mcp 或 MemoMind SDK
from markitdown_mcp import MarkItDownMCP
# 或直接 HTTP 调用 MemoMind REST API
```
