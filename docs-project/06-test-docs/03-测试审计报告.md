## 二、规则合规审计

### 2.1 TDD 铁律

> **规则**：任何新功能、Bug 修复、重构，必须先写失败测试，再写生产代码。

| 模块 | 是否 TDD | 说明 |
|------|---------|------|
| Phase 1 转换器 | ❌ 否 | 先写代码后补测试 |
| Phase 2 MemoMind | ❌ 否 | 先写代码后补测试 |
| Phase 3 测试补充 | ❌ 否 | 先写代码后补测试 |

**结论**：项目整体**未遵循 TDD**。所有模块都是"先代码后测试"（Test-After），不是 TDD。
**改进建议**：Phase 3+ 新功能必须严格执行 TDD。

---

### 2.2 Prove-It 原则

> **规则**：修 Bug 第一件事是写一个能复现它的失败测试。

| 场景 | 是否有复现测试 | 说明 |
|------|--------------|------|
| 编码问题修复 | ❌ | 无复现测试记录 |
| Pipeline 子命令调用 bug | ❌ | 直接修复，无复现测试 |
| GBK 终端编码错误 | ❌ | 直接修复，无复现测试 |

**结论**：未遵循 Prove-It 原则。
**改进建议**：任何 Bug 修复前先写复现测试。

### 2.4 DAMP > DRY 原则

> **规则**：测试代码追求可读性（每个测试自包含讲一个故事），可接受适度重复。

| `test_error_handling.py` | ✅ 优秀 | 测试名即文档，一目了然 |

| `test_cli_options_e2e.py` | ✅ 优秀 | 每个选项独立测试，场景清晰 |
| `test_phase2.py` | ⚠️ 一般 | `_make_doc` helper 可接受，但部分测试名不够描述性 |
| `test_scorer.py` | ⚠️ 一般 | 测试名简洁但可读性尚可 |
